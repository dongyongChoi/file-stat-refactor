# File Stat Refactor

대용량 파일 데이터 기반 통계/분석 기능을 더 안정적이고 확장 가능한 구조로 재설계하는 백엔드 토이 프로젝트입니다.

이 프로젝트는 회사 프로젝트의 코드나 구조를 그대로 옮긴 것이 아닙니다. 기존 회사 프로젝트에서 아쉬웠던 대시보드 통계 생성 방식, 대용량 파일 분석 스캔 처리 방식, SMS 발송 임계치 제어 문제를 바탕으로, 다른 기술 스택과 구조를 사용해 새로 재구현하는 것을 목표로 합니다.

## 기술 스택

| 구분 | 기술 |
| --- | --- |
| Language | Java 21 |
| Framework | Spring Boot 3.5.x |
| Persistence | Spring Data JPA |
| Database | Oracle |
| Messaging | Kafka |
| Change Data Capture | Raw CDC |
| Cache / Lock / Rate Limit | Redis |
| Build | Gradle |
| Test | JUnit 5, Spring Kafka Test |
| Optional | Testcontainers |

## 기존 설계의 문제점과 개선 포인트

이 프로젝트는 기존 회사 프로젝트와 다른 기술 스택으로 기능을 재구현하되, 단순 포팅에 머무르지 않고 운영 상황에서 문제가 될 수 있는 구조를 분리하고 정합성, 성능, 장애 복구 관점에서 다시 설계하는 것을 목표로 합니다.

### 1. 대시보드 진입, 변경기록, 통계 생성 책임의 분산

기존 구조에서는 사용자가 대시보드 화면에 진입하는 시점에 통계 생성 로직이 함께 수행되었습니다. 화면 조회 요청과 무거운 통계 생성 작업이 같은 흐름에 묶이면 트래픽 증가 시 화면 응답이 느려지고, 같은 계정 또는 같은 기준의 통계 생성 요청이 짧은 시간에 중복 실행될 수 있습니다.

또한 업로드, 복사, 이동, 삭제처럼 파일 정보 상태가 변경될 때마다 MyBatis interceptor로 변경기록 테이블에 기록하고, 이후 배치를 통해 통계 데이터에 `MERGE`하는 구조가 있었습니다. 이 방식은 애플리케이션 레벨에서 변경 의도를 기록할 수 있다는 장점이 있지만, 파일 상태를 변경하는 서버가 3개라면 interceptor 설정, 배포, 변경 누락 여부를 모든 서버에서 함께 관리해야 합니다.

개선 방향은 화면 진입 요청과 통계 생성 작업을 분리하고, 파일 변경 감지는 Raw CDC 방식으로 중앙화하는 것입니다. Oracle에 커밋된 파일 정보 변경을 CDC로 감지해 Kafka로 발행하고, Consumer가 변경 이벤트를 통계 갱신 이벤트로 변환해 비동기로 처리합니다. 대시보드 요청은 통계 생성 실행이 아니라 현재 상태와 마지막 생성 결과 조회에 집중하도록 설계합니다.

### 2. 대용량 파일 분석 스캔 기능의 동시 사용 부하

기존 구조에서도 원본 데이터는 정합성을 우선하는 원장 모델로 유지하고, 분석 결과는 별도의 비정규화 테이블에 저장하고 있었습니다. 조회 API 역시 분석 결과 테이블을 기준으로 동작하며, 필터 조건과 정렬 조건에 맞는 복합 인덱스와 페이징 전략이 설계되어 있었습니다.

문제는 중복 파일, 오래된 파일, 대용량 파일 조회를 위해 분석 결과를 생성하거나 갱신하는 스캔 기능입니다. 동시에 많은 사용자가 스캔을 요청하면 같은 계정 또는 같은 조건에 대한 스캔이 반복 실행될 수 있고, 스캔 작업 처리량 증가로 서버와 DB 부하가 커질 수 있습니다.

개선 방향은 스캔 요청 접수와 실제 스캔 실행을 분리하는 것입니다. 스캔 요청은 상태 테이블에 기록하고, 동일 계정과 동일 스캔 조건에 대한 중복 실행은 Redis 선점 로직과 DB Unique 제약으로 제어합니다. 실제 스캔은 비동기 Worker 또는 Kafka Consumer가 처리하고, 사용자는 스캔 상태와 마지막 완료 결과를 조회하도록 설계합니다.

### 3. DB Lock과 `count(*)` 기반 SMS 임계치 체크 비용

초기 구조에서는 SMS 발송이 완료된 뒤 카운트를 증가시키는 방식이었습니다. 이 경우 짧은 시간에 여러 요청이 동시에 들어오면 각 요청이 아직 증가되지 않은 이전 카운트를 기준으로 제한을 통과할 수 있고, 결과적으로 `1분 10건` 제한이 깨질 수 있었습니다.

이 문제는 `SELECT FOR UPDATE`를 사용해 동시에 들어온 요청을 순차적으로 처리하고, 1분 이내 테이블에 insert된 발송 건수를 `count(*)`로 확인하는 방식으로 막고 있었습니다. 정합성 측면에서는 제한을 지킬 수 있지만, 요청마다 DB Lock을 획득하고 건수 조회 쿼리를 수행해야 하므로 트래픽이 증가하면 DB 부하와 대기 시간이 커질 수 있습니다.

개선 방향은 1분 이내 발송 가능 여부 판단을 Redis 원자 연산 기반으로 옮겨 DB Lock과 반복적인 `count(*)` 의존도를 줄이는 것입니다. Redis의 TTL, 카운터, Sorted Set 또는 Lua Script를 활용해 빠르게 제한을 판단하고, DB에는 실제 발송 이력과 감사 가능한 결과를 남기는 구조를 검토합니다.

## 프로젝트 목표

### 1. 대시보드 통계 생성 개선

기존 구조에서는 전통적인 Spring MVC 기반 웹 애플리케이션에서 사용자가 대시보드 페이지에 진입할 때 통계 생성을 수행했습니다. 트래픽이 증가하면 화면 응답 지연, 서버 부하 증가 문제가 발생할 수 있습니다.

이 프로젝트에서는 대시보드 진입 요청과 통계 생성 작업을 분리하고, 파일 변경 감지는 Raw CDC 기반 이벤트 흐름으로 전환합니다.

- 대시보드 요청은 통계 상태와 마지막 생성 결과 조회에 집중합니다.
- Oracle에 커밋된 파일 정보 변경을 Raw CDC로 감지해 Kafka에 발행합니다.
- Consumer가 CDC 이벤트를 통계 갱신 이벤트로 변환하고 비동기로 통계를 생성하거나 갱신합니다.
- MyBatis interceptor 기반 변경기록 로직을 각 서버에 유지해야 하는 부담을 줄입니다.
- 동일 계정 또는 동일 통계 기준에 대한 중복 이벤트와 중복 갱신 작업은 Redis와 DB Unique 제약으로 방지합니다.
- Consumer는 중복 CDC 이벤트나 중복 Kafka 메시지를 받아도 같은 결과를 보장하도록 멱등하게 설계합니다.
- 사용자는 통계 갱신 중 상태와 마지막 생성된 통계를 조회할 수 있습니다.

### 2. 대용량 파일 분석 스캔 처리 개선

현재 구조는 원본 데이터와 분석 결과 테이블이 분리되어 있고, 조회 API는 분석 결과 테이블을 기준으로 동작합니다. 개선 대상은 중복 파일, 오래된 파일, 대용량 파일 분석 결과를 생성하거나 갱신하는 스캔 기능입니다.

이 프로젝트에서는 스캔 요청이 동시에 많이 들어오는 상황에서도 서버와 DB 부하를 제어할 수 있는 구조를 설계합니다.

- 스캔 요청과 실제 스캔 실행을 분리합니다.
- 동일 계정 또는 동일 스캔 조건에 대한 중복 실행을 방지합니다.
- 스캔 상태를 저장해 진행 중, 완료, 실패 상태를 조회할 수 있게 합니다.
- 최근 완료된 분석 결과가 있다면 불필요한 재스캔을 피합니다.
- 분석 결과 조회는 기존 비정규화 테이블, 복합 인덱스, 페이징 전략을 기준으로 유지합니다.

### 3. SMS 발송 임계치 제어 개선

초기 구조에서는 발송 완료 후 카운트를 증가시키는 방식이었기 때문에 짧은 시간에 다수 요청이 동시에 들어오면 `1분 10건` 제한을 우회할 수 있었습니다. 이후 `SELECT FOR UPDATE`로 요청을 순차 처리하고, 1분 이내 발송 이력 테이블의 건수를 `count(*)`로 확인해 제한을 지키는 방식으로 보완했습니다.

이 프로젝트에서는 기존 DB Lock + `count(*)` 방식의 정합성은 유지하되, Redis를 도입했을 때 임계치 판단 비용과 Lock 대기 시간을 얼마나 줄일 수 있는지 검토합니다.

- Redis 기반 Rate Limit 방식을 검토합니다.
- 기존 `SELECT FOR UPDATE` + `count(*)` 방식과 장단점을 비교합니다.
- 동시 요청 상황에서도 `1분 10건` 제한이 정확히 지켜지도록 설계합니다.
- 실패, 롤백, TTL, 재시도, 발송 이력 저장 시점을 함께 고려합니다.
